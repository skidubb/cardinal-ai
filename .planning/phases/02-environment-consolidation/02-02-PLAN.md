# Plan 02: Create consolidated root .env, .env.example, and update docker-compose.yml

## Objective
Create the single root `.env` file by merging all 3 per-project `.env` files, create a comprehensive `.env.example` documenting all keys, standardize naming inconsistencies (LANGFUSE_BASE_URL not LANGFUSE_HOST, NOTION_API_KEY not NOTION_TOKEN), and update `docker-compose.yml` to use `${VAR}` interpolation from root `.env` instead of hardcoded credentials.

## must_haves
- Root `.env` contains the superset of all keys from Agent Builder, Orchestration, and Evals `.env` files
- Key naming is standardized: `LANGFUSE_BASE_URL` (not HOST), `NOTION_API_KEY` (not TOKEN)
- Root `.env.example` documents every key with: comment describing purpose, which projects use it, whether required or optional
- `.env.example` groups keys by category (LLM, Observability, Storage, Search, Integration, Config, Docker)
- `docker-compose.yml` uses `${POSTGRES_PASSWORD}` (no default) for password, `${POSTGRES_USER:-ce}` and `${POSTGRES_DB:-ce_platform}` with safe defaults
- Metabase service uses `${POSTGRES_USER:-ce}` and `${POSTGRES_PASSWORD}` (referencing same vars as postgres service)
- Root `.env` includes `POSTGRES_USER=ce`, `POSTGRES_PASSWORD=ce_local`, `POSTGRES_DB=ce_platform`, `POSTGRES_HOST=localhost`, `POSTGRES_PORT=5432`
- Root `.env` is in `.gitignore`; `.env.example` is tracked
- Commented-out langfuse section in docker-compose.yml also uses `${POSTGRES_*}` vars if present
